Context Engineering Is the New #1 Skill

Foundation · Replaces prompt engineering entirely

Karpathy, June 2025

"Context engineering is the delicate art and science of filling the context window with just the right information for the next step."

CE-01 The 4 core operations to master Foundation ▾
SELECT — only pull what's needed
Use RAG, AST parsing, grep, and semantic similarity to inject relevant code and docs. Providing all tools simultaneously hurts accuracy 3× vs semantic selection. The context window is scarce — treat it like memory in a performance-critical system.
Tool selection via semantic similarity = 3× better accuracy vs giving Claude all tools at once
WRITE — persist knowledge across sessions
Use scratchpads, tasks/lessons.md, and long-term memory files. Boris Cherny's rule: "Anytime Claude does something incorrectly, add it to CLAUDE.md so it never repeats." Every correction encoded compounds indefinitely.
COMPRESS — compact at 70–95% context usage
Don't wait for auto-compact. Proactively clear with /clear between unrelated tasks. Compact at 70% context usage as a habit. Fresh context produces measurably better output — session hygiene is part of the craft.
/clear between unrelated tasks — treat sessions like git branches
ISOLATE — give each agent its own context window
Multi-agent isolation delivers 90.2% higher task success vs single-agent execution. The cost is 15× tokens but the accuracy gain justifies it for any non-trivial work. Each agent gets its own tools, instructions, and memory — no cross-contamination.
Isolation = 90.2% higher success rate (15× token cost — worth every token)
CE-02 The context budget allocation strategy Advanced ▾
Treat the context window like memory management in high-performance computing. Allocate by priority, not by convenience.

Plan Before You Build

Workflow · Non-negotiable for any task touching 3+ files

PL-01 Plan Mode is not optional for complex tasks Core Habit ▾
Start every 3+ file task in Plan Mode
Boris Cherny starts most sessions in Plan Mode. The spec you write upfront is product thinking made explicit — a vague spec multiplies errors across your entire agent fleet. The more autonomous your agents, the more expensive a bad spec becomes.
Spawn a second Claude to review your plan
Before executing, open a second session and ask it to review your plan as a staff engineer. Cross-model review catches architectural drift before any code is written. This costs minutes and saves hours.
"Review this plan as a staff engineer. What's wrong, risky, or missing?"
Phase-wise gated execution
Break tasks into atomic phases of 5–10 minutes each. Each phase must pass unit, automation, and integration test gates before proceeding. No phase skipping, no exceptions. Build the gates into the plan itself, not as an afterthought.
If Claude goes sideways — stop, don't fix forward
Use Esc+Esc (/rewind) when Claude drifts. If you've corrected the same mistake twice, clear and restart. Correcting in-context piles debt on top of wrong assumptions — you're not fixing the problem, you're burying it.
After any mediocre fix: "Scrap this. Implement the elegant solution from scratch."
03

Delegate Aggressively, Verify Relentlessly

Execution · The quality gap is usually an accountability gap

DV-01 The delegation and verification system Execution ▾
Delegate what's verifiable — hold what's design-critical
Engineers delegate tasks they can "sniff-check for correctness" quickly: boilerplate, tests, migrations, formatting, documentation. Keep conceptually difficult or architecture-defining work collaborative. Your judgment is the scarce resource — spend it wisely.
Build feedback loops into CLAUDE.md permanently
Tell Claude to run tests, linters, and build commands after every change. Without a feedback loop, Claude assumes changes work and moves on. With one, it self-corrects. Boris says this alone can 2–3× output quality — it's the single highest-leverage CLAUDE.md addition.
Always run pytest after changing Python files → baked into CLAUDE.md, not a one-time prompt
You are the final approval gate — never skip it
"It compiled, ship it" is the most expensive mistake in agentic engineering. Every PR gets human review. Develop pattern recognition for agent-generated code: common failure modes, architectural drift, security holes, over-engineering.
Challenge Claude — don't just accept it
Use adversarial verification prompts that force Claude to defend its output. The quality gap is often just an accountability gap — Claude will produce better work when it knows it has to justify itself.
"Grill me on these changes" · "What would a staff engineer say is wrong here?" · "Prove to me this works."
04

Git Discipline = Agentic Safety Net

Safety · One worktree per session is the baseline

GD-01 Worktrees, branches, and squash merges Safety ▾
One git worktree per Claude session
Boris runs 10–15 Claude sessions simultaneously, each in its own git worktree. Changes don't collide. You can review, merge, or discard independently. This is not an advanced technique — it's the team's baseline for productive multi-session work.
git worktree add ../feature-x && cd ../feature-x && claude
Always create a branch before Claude touches code
One task per branch. Never let agents work directly on main. Small, focused PRs: the Anthropic team's median is 118 lines across 141 PRs, with 45K lines changed in a single day. Small PRs = fast reviews = fast feedback loops.
Squash merge everything from agent branches
Squash merging gives you a clean linear history, one commit per feature, easy git revert and git bisect. Agent-generated intermediate commits are noise — they reflect the model's thinking process, not your project's history. Squash them away.

CLAUDE.md Mastery

Infrastructure · As essential as .gitignore

Community Consensus 2026

CLAUDE.md is now as important as .gitignore — it's essential infrastructure, not optional docs. Every project needs one. It's the permanent brain that survives session resets.

CM-01 The 100-line rule (Boris Cherny's secret) Structure ▾
Boris's CLAUDE.md is ~100 lines. Most devs write 500–1000. Boris's works better.
Every line must earn its place. For each line ask: "Would removing this cause Claude to make mistakes?" If not, cut it. Bloated files cause Claude to ignore your actual instructions — critical rules get lost in the noise.
The 10-section structure that works
Project overview · Tech stack · Architecture · Code standards · Key file locations · Common commands · Workflow rules · Gotchas/quirks · Lessons learned · Agent delegation rules. Each section earns its place by preventing a specific class of mistake.
Use @imports for detailed sections — not inline walls of text
Root CLAUDE.md stays 50–100 lines with @imports for sub-sections. Domain knowledge and niche workflows live in child CLAUDE.md files per directory — Claude loads them on demand without bloating every session.
@./docs/auth-patterns.md — loaded only when working in the auth module
Treat it like code: version, review, prune regularly
Check it into git. Never add to .gitignore — it's documentation AND institutional knowledge AND onboarding material for new developers. Review it when things go wrong. Update it multiple times a week as the codebase evolves.

The Self-Improvement Loop

Compounding gains · Every correction becomes a permanent rule

SI-01 Boris's 6-step compounding loop Compounding ▾
The loop: Plan → Execute → Review → Correct → Write rule → Import into CLAUDE.md
Each correction you encode makes every future session better. This is compounding engineering — the kind that builds a real competitive moat over time. After 6 months of consistent looping, your CLAUDE.md knows your codebase better than most new hires.
After any correction: "Write a rule in tasks/lessons.md that prevents this mistake."
The 3 principles at the bottom of Boris's file
1. Make every change as simple as possible. Delete lines, don't add them. 2. Fix root causes — no band-aids, no temporary fixes. 3. Only touch what's necessary — no side effects, no scope creep. These 3 lines prevent more bugs than any linter.
Use "IMPORTANT" and "YOU MUST" for critical rules
Emphasis works. If Claude keeps violating a rule despite it being in CLAUDE.md, the file is either too long (rule gets lost) or the phrasing is ambiguous. Add emphasis or move the rule to the top. Don't tolerate repeated violations — encode them away.

Multi-Agent Systems

Architecture · The mental model shift from conductor to orchestrator

The Paradigm Shift

You are no longer the lead engineer. You talk to a lead agent. In 2026, top engineers stop talking to individual agents and start talking to an orchestrator that conducts the fleet. You design the system and verify the output.

MA-01 Old model vs new model Core concept ▾
Old Model
You → Single agent → Output (sequential, context-capped, all-or-nothing)
New Model
You → Orchestrator → [Frontend agent | Backend agent | Security agent | Test agent] → Parallel output → Synthesis → Verified PR
Strong engineers get MORE leverage, not less
A vague spec multiplies errors across dozens of parallel runs. A precise spec — with clear architecture, integration boundaries, edge cases, and invariants — multiplies into precise implementations everywhere. Your value shifts from typing speed to specification quality.

The 4 Production Orchestration Patterns

Architecture · Choose the right topology for the task

OP-01 All 4 patterns explained Architecture ▾
1. Pipeline — sequential stages, highest accuracy
Research agent → Draft agent → Review agent. Each output becomes the next input. Use for content generation, document processing, multi-step transformations. Tradeoff: more latency, highest accuracy. Best when correctness matters more than speed.
2. Orchestrator-Worker — parallel execution, 90.2% higher success
One orchestrator decomposes the task and delegates to specialist workers in parallel, then synthesizes results. Best for codebases with clear module boundaries. Start with 1 orchestrator + 1 worker. Add agents only when you have evidence of a bottleneck.
Start small: 1 orchestrator + 1 worker. Add agents only with evidence, not intuition.
3. Router — classify and dispatch
A triage agent classifies the request type and routes to the specialist: Bug → Debug agent, Feature → Architecture agent, Refactor → Refactoring agent, Question → Research agent. Keeps each specialist's context clean and focused.
4. Plan-and-Execute — 90% cost reduction
Frontier model (Claude Opus) creates the strategy. Cheaper model (Claude Haiku) executes the steps. Reduces cost by 90% vs running Opus for everything. Model tiering is now a standard architectural decision, not an optimization afterthought.
The merge rule
If two agents spend more than 80% of their time on similar tasks → merge them. If one agent's context is consistently polluted by unrelated responsibilities → split it. Don't over-engineer the topology. Start simple, split only when necessary.

Addy Osmani's framework · Use all three tiers

TS-01 Which tier for which work Strategy ▾
Tier 1
Single terminal, interactive. Claude Code subagents, Claude Squad. Start here. Best for exploratory work and 3–5 agent teams on known codebases. You're in control and watching in real-time.
Tier 2
Local multi-session, dashboard-driven. Conductor, Vibe Kanban, Gastown. Your machine, multiple worktrees, diff review + merge control. Best for 3–10 agent parallel sprints where you need oversight but want parallelism.

Tier 1 for interactive work, Tier 2 for parallel feature sprints, Tier 3 for background tasks and tech debt. Don't outsource your mastery of agents to tools — understanding the coordination patterns matters more than any specific tool you use.

MCP + A2A: The New HTTP for Agents

Protocols · The connectivity layer that makes agent fleets possible

MP-01 MCP and A2A explained Protocols ▾
MCP is the plug-and-play layer for tool connectivity
Model Context Protocol standardizes how agents connect to external tools, databases, and APIs. Custom integration work becomes plug-and-play. Use lazy loading — tools activate only when context triggers them. 78% startup cost savings with lazy MCP loading.
A2A enables cross-agent collaboration at scale
Google's Agent-to-Agent Protocol lets agents from different vendors and platforms communicate peer-to-peer. You can now compose agents from standardized components rather than building monolithic proprietary systems. This is the shift from islands to infrastructure.
Production MCP config: always apply these 3 safety settings
Set read_only: true for all production data MCP servers. Use lazy_load with triggers so tools don't bloat context on every session. Block sensitive path patterns (*.env, *.key, */secrets/*) at the config level — not as a prompt-time reminder.
The circuit breaker rule: error rate > 1% = automatic halt. No exceptions.
// Production MCP config { "postgres": { "read_only": true, "lazy_load": true, "triggers": ["database", "sql", "schema"], "blocked_patterns": ["*.env", "*.key", "*/secrets/*"] }, "github": { "lazy_load": true, "triggers": ["git", "pr", "commit", "branch"] } }

Security & Safety in Agentic Systems

Non-negotiable · The blast radius you don't see is the one that hurts

Critical 2026 Finding

Agentic systems operate with broad permissions and can take irreversible actions. The #1 cause of agentic incidents in 2026 is under-scoped permission boundaries — not model errors. Security must be designed in, not bolted on after the fact.

SS-01 The 5 security primitives every agentic system needs Non-Negotiable ▾
Principle of Least Privilege — always
Every agent gets only the permissions required for its current task — nothing more. Read-only access for research agents. Write access scoped only to sandboxed directories. Never grant production database write access unless the task explicitly requires it and a human approves it. Default stance: read-only. Escalate with explicit justification logged.
MCP read_only: true for all production data sources. Block *.env, *.key at config level — not as a prompt reminder.
Blast Radius Minimization
Design every agent workflow so the worst-case failure affects the smallest possible surface area. Use isolated git branches, sandboxed environments, and ephemeral containers. Never allow an agent's error to cascade into production. The isolation you build upfront is insurance you'll never regret paying for — because the one time you skip it will cost more than all the times you didn't.
Human-in-the-Loop Checkpoints
Hardcode approval gates before any irreversible action: file deletion, production database writes, external API calls with cost implications, or any code pushed to main. Define your irreversibility threshold upfront — anything above it requires explicit human sign-off. These checkpoints are not friction; they are what makes autonomous systems trustworthy.
Rule: any action that cannot be undone in <60 seconds requires human confirmation before execution.
Prompt Injection Protection
When agents read from external sources — web pages, files, APIs, user input — malicious content can inject instructions directly into the agent's context. Always sanitize external data before injection. Never pass raw user input to an orchestrator. Use typed schemas for agent-to-agent communication, not freeform text. Treat all external content as untrusted input, because it is.
Secret Management — never in the context window
API keys, credentials, and secrets must never appear in the context window — they get logged, cached, and can be leaked. Use environment variables and dedicated secret managers (AWS Secrets Manager, HashiCorp Vault, 1Password CLI). Block sensitive patterns at the MCP config layer, not as a conversational instruction Claude can forget.
Use a secrets manager CLI, not .env files in the project root. Never mention a secret value in a prompt — even to explain it.
SS-02 Audit trails and circuit breakers for production agents Production ▾
Log every agent action with its justification
Every file touched, command run, and API call made should be logged with the agent's stated reasoning. This is your post-mortem trail. Without it, debugging a multi-agent failure is archaeology. Use structured JSON logging. In regulated environments, store logs with tamper-evident checksums and a 90-day retention minimum.
The circuit breaker pattern
If an agent exceeds a defined error rate — e.g., >1% of operations fail — halt the entire workflow automatically. Don't let a failing agent spiral into a cascading fleet failure. Circuit breakers belong in the orchestrator as a first-class architectural feature, not as an emergency afterthought. Build them before you need them.
Circuit breaker threshold: error rate >1% = automatic halt + alert + rollback. No exceptions, no override flags.
// Minimal agent audit log schema (JSON) { "ts": "2026-04-27T14:23:01Z", "agent_id": "backend-worker-3", "action": "file.write", "path": "src/auth/middleware.ts", "justification": "Adding rate-limit guard per spec PL-01", "approved_by": "orchestrator", "reversible": true, "rollback_cmd": "git checkout src/auth/middleware.ts", "session_id": "ses_abc123", "error_rate": 0.000 }

Tools & Stack

Decision matrix · Right tool for the right job

The Golden Rule

Don't outsource your mastery of agents to tools. Understand the coordination patterns first, then pick the tool. A well-designed 2-agent system beats a poorly designed 10-agent one every time.

TL-01 When to use which tool Decision Matrix ▾
Claude Code
Long autonomous tasks, multi-agent orchestration, codebase-wide refactors, MCP integration, anything requiring sustained multi-step execution. Your primary agentic terminal.
GitHub Copilot
Inline autocomplete, quick single-function generation, IDE-native flow, pair programming feel. Complementary to Claude Code — they don't overlap, they compound.
Cursor
Interactive in-editor agent mode, fast iteration on focused files, visual diff review. Best for synchronous work where you want to see changes happen in context.
Codex / Jules
Cloud agents for background task execution. Fire-and-forget: assign a PR-ready task, return to a diff. Best for draining backlog items overnight without burning compute hours interactively.
The two-tool stack that works best
Copilot handles inline autocomplete in the IDE. Claude Code handles everything agentic in the terminal. They don't overlap — they complement. This is the most common high-productivity setup in 2026. Start here before adding complexity.
TL-02 Multi-agent frameworks When You Need More ▾
LangGraph
Graph-based workflows with typed state. Best for complex branching, loops, and visual monitoring of agent transitions. Most searched framework in 2026.
CrewAI
Role-based agents with distinct responsibilities. Best when you need agents that mirror how a human team works. Great for content and research workflows.
AutoGen / Semantic Kernel
Enterprise-grade orchestration with OpenTelemetry tracing and self-healing properties. Use when you need full observability and compliance-grade audit trails.
Claude SDK
Native Anthropic agent framework. Tightest integration with Claude models, MCP, and A2A. Best if you're building Claude-native products from the ground up.
Model tiering for 40–60% cost reduction
Mixing models reduces costs 40–60% vs running one premium model everywhere. Claude Haiku for triage/routing, Claude Sonnet for standard tasks, Claude Opus for complex reasoning and orchestration. This is now a standard architectural decision.

Mindset Shifts That Unlock 100X

Mental models · The questions that change everything

"Am I writing code or designing systems?"
The most valuable engineers in 2026 aren't those who type the fastest. They're those who design coordination systems. Shift your identity from code writer to system designer. Your leverage is in what you architect, not what you type.
"Is this task verifiable quickly?"
If yes → delegate fully and aggressively. If not → work collaboratively with Claude. The delegation intuition is a skill you build over time. Start by delegating anything with clear acceptance criteria.
"Am I treating agents as leverage or magic?"
Treat agents as leverage, not magic. Use them aggressively where speed is cheap. Be conservative where mistakes are expensive. The 80% problem — AI solutions that are almost right — comes from skipping verification.
"Is my codebase agent-ready?"
Agents struggle with disorganized codebases, unclear conventions, and missing documentation. A well-structured project enables dramatically better agent performance. Investment in code organization compounds as you scale agents.
"What would a staff engineer say is wrong here?"
The single best quality-gate prompt. Ask it after every significant agent output. Build it into your slash commands. This is Boris Cherny's verification standard — never mark a task complete without passing it.
"Am I compounding or just consuming?"
The self-improvement loop is the difference between linear and exponential productivity. Every session where you don't encode a lesson is a session that didn't compound. After 6 months of looping, your CLAUDE.md is a serious competitive moat.

10 Anti-Patterns That Kill Agentic Velocity

Failure modes · Recognise them before they compound into real damage

The Hidden Tax

Most developers unknowingly sacrifice 40–60% of their agentic productivity to just 3 of these patterns: not encoding lessons, skipping planning, and over-trusting output. Recognising them is the first step to eliminating them permanently.

01
Context Window Hoarding
Running the same session for hours, accumulating irrelevant context. Each distinct task deserves a fresh, focused window. Use /clear aggressively — treat sessions like git branches.
02
Over-trusting Agent Output
"It compiled, ship it." Every PR requires human review. Build pattern recognition for agent failure modes: silent scope creep, confident hallucination, over-engineering of simple problems.
03
Building Monolithic Agents
One agent that handles frontend + backend + tests + docs. It fails at all of them. Specialist agents with isolated contexts dramatically outperform single-context generalists on every measurable metric.
04
Skipping the Planning Phase
Jumping straight to execution on a 3+ file task. A bad spec costs 10× more to fix post-execution than pre-execution. Plan Mode is non-negotiable. The spec is the product, not the overhead.
05
Not Encoding Corrections
Correcting Claude verbally in the session, then repeating the same correction the next day. Every correction not written to CLAUDE.md is a correction that doesn't compound into your permanent advantage.
06
Agents Working on Main
Letting Claude work directly on main or staging. One branch per task, one worktree per session — always. Worktrees are free to create. Rollbacks from production incidents are not.
07
Opus for Everything
Using your most powerful and expensive model for routing, triage, and formatting. Model tiering saves 40–60% in cost with zero accuracy loss on appropriate tasks. Haiku for triage, Sonnet for standard tasks, Opus for complex reasoning only.
08
No Feedback Loops in CLAUDE.md
A CLAUDE.md that doesn't tell Claude to run tests after changes. Without this loop, Claude assumes changes work and moves on. A single "run pytest after every Python edit" line can 2–3× output quality alone.
09
Fuzzy Agent Boundaries
Agents with vague responsibilities like "handle the backend." Without crisp integration contracts — types, schemas, API boundaries — agents generate code that conflicts with each other's silent assumptions.
10
Premature Orchestration Complexity
Building 10-agent systems before validating that a 2-agent solution doesn't solve the problem. Every agent you add multiplies coordination overhead and failure surface. Start minimal. Expand only with performance evidence, never intuition.
